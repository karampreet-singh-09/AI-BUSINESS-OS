# Deployment Guide

We use Vercel for frontend/API deployment and Supabase Cloud for database and Auth.

## Environments
1. **Preview**: Automatically generated by Vercel for every Pull Request.
2. **Staging**: The `main` branch. Deploys to a dedicated staging environment connected to a staging Supabase project.
3. **Production**: Triggered by a specific release tag or manual promotion in Vercel. Connected to the production Supabase project.

## Database Migrations
Migrations must be strictly managed to avoid data loss.
1. Local development changes are captured in `supabase/migrations/`.
2. When a PR is merged to `main`, a GitHub Action automatically applies the migrations to the Staging database.
3. Upon Production release, the GitHub Action applies migrations to the Production database BEFORE the Vercel build completes.

## Vercel Configuration
- **Framework**: Next.js
- **Build Command**: `npm run build`
- **Output Directory**: `.next`
- **Environment Variables**: Managed directly in the Vercel dashboard. Ensure all secrets match the required variables in `ENVIRONMENT_SETUP.md`.

## Rollback Procedure
If a production deployment causes a critical failure:
1. In Vercel, navigate to the Deployments tab and click "Redeploy" on the previous stable version.
2. If the failure is database-related, evaluate if a reverse-migration script is necessary, or restore from Supabase Point-in-Time Recovery (PITR). (Note: Reverting code is easy; reverting DB schemas requires extreme caution).
